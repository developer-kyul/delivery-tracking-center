# 🚚 README 필수 항목

## 1. 디자인 결정 사항

### ✅컴포넌트 구조 분리 이유

컴포넌트 분리 기준은 "**역할과 책임**이 다른가?" 를 가장 먼저 고려했습니다.

검색 및 필터 영역, KPI 통계 영역, 배송 목록 영역은 각각 독립적인 역할을 가지고 있기 때문에 별도 컴포넌트로 분리했습니다.  
또한 레이아웃, 모달, 페이지네이션처럼 재사용 가능성이 높은 요소도 별도 컴포넌트로 구성했습니다.

이 구조를 통해 유지보수성을 높이고, 향후 검색 조건 추가, KPI 항목 변경, 목록 UI 개선 등이 필요할 때 각 영역을 독립적으로 수정할 수 있도록 설계했습

### ✅상태 관리 방식 선택 이유

상태 관리 방식은 프로젝트 규모와 상태 공유 범위를 고려하여 React의 **useState**를 선택했습니다.

검색어, 상태 필터, 화물 종류 필터 등은 배송 현황 페이지 내부에서만 사용되는 로컬 상태였기 때문에 전역 상태 관리 라이브러리를 도입할 필요가 없다고 판단했습니다.

또한 상태를 상위 컴포넌트에서 관리하고 하위 컴포넌트에 Props로 전달하는 단방향 데이터 흐름 구조를 적용하여 상태 변경 추적을 단순하게 유지했습니다.

전역 상태 관리 도구를 사용할 수도 있었지만 현재 규모에서는 오히려 복잡도가 증가한다고 판단했습니다. 상태는 상위 컴포넌트에서 관리하고 하위 컴포넌트에 전달하는 구조로 데이터 흐름을 단순하게 유지했습니다

### ✅색상 및 타이포그래피 결정 근거

#### 🎨 색상 선택 근거

색상은 **신호등 색상 체계**에서 착안하였습니다.일상적으로 익숙한 색상 의미를 활용하여 별도의 설명 없이도 상태를 직관적으로 인지할 수 있도록 설계하였습니다.

- 배송 준비중 : 회색
- 배송중 : 파란색
- 배송완료 : 초록색
- 배송지연 : 주황색
- 반송 : 빨간색

또한 전체 화면은 연한 색상 사용을 사용하고, 상태 정보에만 강조 색상을 적용하여 사용자가 중요한 정보에 집중할 수 있도록 구성하였습니다.

#### ✍️ 타이포그래피 선택 근거

프로젝트의 기본 폰트는 **Noto Sans KR**을 사용하였습니다.

Noto Sans KR은 한글 가독성이 우수하고 다양한 운영체제 및 브라우저 환경에서 안정적으로 표시되며, 기업용 대시보드와 같은 정보 중심 서비스에 적합하다고 판단하였습니다.

Pretendard도 고려하였으나, 보다 중립적이고 안정적인 인상을 주는 Noto Sans KR이 물류 관리 시스템의 특성과 더 적합하다고 판단하여 최종 선택하였습니다.

## 2. 트레이드 오프(Trade-off)

### ✅반응형 미구현

| 구분    | 내용                                   |
| ------- | -------------------------------------- |
| ⭐ 선택 | Desktop 환경에 최적화된 적응형 UI 구현 |
| ❌ 포기 | 모바일 반응형 지원                     |

물류 대시보드는 한 화면에 표시해야 하는 정보의 밀도가 높기 때문에, 단순히 화면 크기에 따라 레이아웃을 축소하는 반응형 방식으로는 **정보 전달력과 사용성이 저하될 수 있다고 판단**하였습니다. 특히 상세 정보를 팝업 형식으로 제공하는 현재 구조는 데스크탑 환경에 최적화되어 있어, 모바일에서 그대로 적용할 경우 사용성이 더욱 저하될 수 있다고 판단하였습니다.

이에 따라 제한된 개발 기간 내에는 배송 상태 조회, 검색 및 필터링, KPI 시각화 등 핵심 기능 구현에 집중하였으며, 파비콘 및 OG 메타 태그 설정, 마이크로 인터렉션 등 세부적인 사용자 경험 요소도 함께 반영하였습니다.

## 3. 어려웠던 점

### ✅배송 상태 데이터 구조 설계

배송 시스템은 준비중 → 배송중 → 완료와 같은 정상 흐름뿐 아니라 배송지연, 반송과 같은 예외 상황도 함께 고려해야 했습니다.

초기에는 모든 상태를 하나의 순차적인 흐름으로 표현하려 했지만, 배송지연과 반송은 특정 단계가 아닌 배송 과정에서 발생하는 예외 상태라고 판단했습니다.

이를 해결하기 위해 **정상 배송 흐름과 예외 상태를 분리하여 설계**하였으며, 상태 코드·라벨·스타일 정보를 상수로 관리하여 화면과 데이터를 분리했습니다.

그 결과 상태 관리 로직을 단순화할 수 있었고, 향후 실제 API 연동 시에도 유연하게 확장할 수 있는 구조를 구성할 수 있었습니다.
